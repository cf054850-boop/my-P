Live Server

מבוא ל-Live Server

Live Server הוא שרת מקומי קל-משקל המיועד לפיתוח Front-End. הוא מאפשר לפתוח קבצי HTML בסביבת שרת אמיתית מהמחשב, ומרענן את הדפדפן אוטומטית בכל שמירת קובץ.

בלי Live Server

  • פתיחת קובץ ישירות מהמחשב (file://)
  • קריאת קבצים חיצוניים חסומה (CORS)
  • fetch לא עובד על קבצים מקומיים
  • רענון ידני אחרי כל שינוי

עם Live Server

  • הדף נטען דרך http://localhost
  • קריאת JSON ומשאבים מקומיים אפשרית
  • fetch פועל כצפוי
  • רענון אוטומטי בשמירה
השוואה ל-C#:
ב-C# ASP.NET: IIS Express מריץ שרת מקומי אוטומטית בעת הרצת הפרויקט.
ב-JS Front-End: Live Server ממלא תפקיד דומה — מספק סביבת שרת לפיתוח מקומי.
שקופית 4 מתוך 15
localhost

localhost ופורט 5500

כאשר Live Server פועל, ניתן לגשת לפרויקט דרך כתובת כגון:

http://127.0.0.1:5500/index.html // או בשם: http://localhost:5500/index.html

פירוק הכתובת

  • http:// — הפרוטוקול. הדפדפן מתקשר עם השרת כ-HTTP רגיל
  • 127.0.0.1 / localhost — כתובת ה-loopback: "המחשב הזה עצמו". אינה יוצאת לאינטרנט
  • :5500 — הפורט: ה"דלת" שדרכה שרת Live Server מאזין. ניתן לשינוי בהגדרות
  • /index.html — הנתיב לקובץ בתוך תיקיית הפרויקט
פורט הוא מספר שמאפשר להריץ מספר שרתים על אותו מחשב בו-זמנית. כל שרת מאזין על פורט שונה. Live Server משתמש ב-5500 כברירת מחדל, אך אם הפורט תפוס — הוא יעבור ל-5501, 5502 וכן הלאה.
השוואה ל-C#:
IIS Express ב-Visual Studio מריץ שרת על פורט כגון :44300.
הרעיון זהה — שרת מקומי על פורט ייחודי.

אסינכרוניות

מבוא לאסינכרוניות

JavaScript היא שפה חד-תהליכית (single-threaded) — רק פעולה אחת מתבצעת בכל רגע. כדי שפעולות ארוכות (כגון קריאת שרת) לא יקפיאו את כל הממשק, JavaScript מבצעת אותן אסינכרונית — ממשיכה לרוץ, ומטפלת בתגובה כשתגיע.

סינכרוני (Synchronous)

  • שורה אחר שורה, בסדר
  • כל שורה מחכה לסיום הקודמת
  • פעולה ארוכה — חוסמת הכל
  • כמו תור בסופרמרקט

אסינכרוני (Asynchronous)

  • פעולה מתחילה, קוד ממשיך
  • תגובה מטופלת כשתגיע
  • לא חוסם את הממשק
  • כמו הזמנת אוכל — ממשיכים לשבת
console.log("1 — לפני הבקשה"); fetch("./data.json") .then(res => res.json()) .then(data => { console.log("3 — הנתונים הגיעו:", data); }); console.log("2 — אחרי הבקשה (לפני שהנתונים הגיעו!)"); // סדר הפלט: 1 → 2 → 3
השוואה ל-C#:
ב-C#: async/await עם Task — אותו עיקרון של אי-חסימה.
ב-JS: Promise עם .then() הוא המנגנון המסורתי. גם ב-JS קיים async/await.

Request / Response

כיצד הדפדפן עובד עם קריאות שרת

כל קריאה ל-fetch עוברת מחזור של בקשה-תגובה (Request-Response). מכיוון שהרשת אינה מיידית, התגובה מגיעה לאחר זמן לא ידוע, ולכן לא ניתן להמתין לה בצורה סינכרונית.

// מה שקורה כשכותבים: fetch("./flights.json") // 1. הדפדפן שולח בקשת HTTP GET לשרת // 2. הדפדפן לא מחכה — ממשיך לשורה הבאה // 3. בשלב מסוים, השרת מחזיר תגובה (Response) // 4. הדפדפן "מעיר" את ה-.then() הראשון עם ה-Response // 5. result.json() מפרסר את גוף התגובה — גם זה אסינכרוני! // 6. ה-.then() השני מקבל את הנתונים הסופיים
מדוע .then() מקבל פונקציה?
כי לא ידוע מתי התגובה תגיע. במקום "לחכות" (וחסום), רושמים פונקציה שתופעל כשהנתונים יהיו מוכנים. זהו עיקרון ה-callback שעליו מבוסס כל מנגנון ה-Promise.
השוואה ל-C#:
HttpClient.GetAsync(url).ContinueWith(task => { ... }) — אותה זרימה: שלח בקשה, המשך, טפל כשתגיע.
await ב-C# וב-JS מסתיר את ה-callback ומהפך את הקוד לקריא יותר.

DevTools

צפייה בנתונים בדפדפן — DevTools

כלי הפיתוח של הדפדפן (F12) מכילים לשוניות ייעודיות למעקב אחרי קריאות רשת, נתונים שמורים ותוצאות JSON.

Network Tab

  • כל בקשות ה-HTTP שנשלחו
  • סטטוס התגובה (200, 404, 500)
  • גוף התגובה ב-JSON
  • זמן הבקשה וגודלה
  • Headers של הבקשה והתגובה

Application Tab

  • צפייה ב-localStorage
  • צפייה ב-sessionStorage
  • עריכה ומחיקה ידנית
  • Cookies
  • Cache Storage

שימוש אופייני בפיתוח

// לאחר קריאת fetch: // 1. פתיחת DevTools (F12) // 2. לשונית Network → לחיצה על בקשת ה-JSON // 3. לשונית Preview → צפייה במבנה הנתונים // לאחר שמירה ב-localStorage: // 1. לשונית Application → Local Storage → origin // 2. צפייה בכל מפתחות הערך שנשמרו
כאשר fetch אינו עובד, הצעד הראשון הוא תמיד בדיקת לשונית Network. קוד תגובה 200 = הצלחה, 404 = הקובץ לא נמצא, 0 / Failed = בעיית CORS או שרת לא פועל.

fetch

מבוא ל-fetch

fetch היא פונקציה מובנית בדפדפן לשליחת בקשות HTTP. היא מחזירה Promise — אובייקט המייצג ערך שיהיה זמין בעתיד.

// מבנה בסיסי: fetch(url) .then(response => { /* טיפול בתגובה */ }) .catch(error => { /* טיפול בשגיאה */ }); // דוגמה פשוטה: fetch("./products.json") .then(response => response.json()) // המרת התגובה ל-JavaScript .then(data => { console.log(data); // שימוש בנתונים }) .catch(err => { console.error("שגיאה:", err); });
fetch אינו זורק שגיאה על תגובת HTTP שגויה (404, 500). הוא רק נכשל אם הבקשה לא הגיעה כלל (שגיאת רשת, CORS). לכן יש לבדוק ידנית את response.ok ולזרוק שגיאה במידת הצורך.
השוואה ל-C#:
new HttpClient().GetAsync(url) — מחזיר Task<HttpResponseMessage>.
ב-JS: fetch(url) — מחזיר Promise<Response>. אותו רעיון, אחרת ממומש.

JSON

עבודה עם JSON

JSON (JavaScript Object Notation) הוא פורמט טקסטואלי לשמירה והעברה של נתונים מובנים. הוא שפה ניטרלית — ניתן לקרוא ולייצר JSON בכל שפת תכנות.

אובייקט JavaScript

const flight = { id: 1, from: "TLV", to: "LHR", seats: 42 };

מחרוזת JSON

'{ "id": 1, "from": "TLV", "to": "LHR", "seats": 42 }'
// אובייקט JS → מחרוזת JSON (לשמירה/שליחה): const str = JSON.stringify(flight); // '{"id":1,"from":"TLV","to":"LHR","seats":42}' // מחרוזת JSON → אובייקט JS (לשימוש): const obj = JSON.parse(str); console.log(obj.from); // "TLV" // הבדל מהותי: response.json() לעומת JSON.parse(): // response.json() — ממיר את גוף התגובה מ-HTTP Response // JSON.parse() — ממיר מחרוזת טקסט רגילה

קובץ JSON

קריאת קובץ JSON מתוך הפרויקט

ניתן לאחסן נתונים בקובץ .json בתוך הפרויקט ולטעון אותם באמצעות fetch. פעולה זו דורשת שרת מקומי — כלומר Live Server חייב לפעול.

מבנה קובץ flights.json

{ "flights": [ { "id": 1, "from": "TLV", "to": "LHR", "price": 450 }, { "id": 2, "from": "TLV", "to": "CDG", "price": 320 }, { "id": 3, "from": "TLV", "to": "JFK", "price": 680 } ] }

קריאת הקובץ

// נתיב יחסי לתיקיית הפרויקט: fetch("./flights.json") .then(response => response.json()) .then(data => { console.log(data.flights); // מערך הטיסות data.flights.forEach(f => { console.log(f.from + " → " + f.to); }); });
פתיחת קובץ ה-HTML ישירות מהמחשב (file://) תגרום ל-fetch להיכשל עם שגיאת CORS. יש תמיד לפתוח את הפרויקט דרך Live Server (http://localhost:5500).

localStorage

localStorage — שמירת נתונים בדפדפן

localStorage מאפשר שמירת נתונים קבועה בדפדפן — הנתונים נשארים גם לאחר סגירת הדפדפן ורענון הדף, עד שנמחקים ידנית.

// שמירה (מחרוזות בלבד — אובייקט דורש stringify): localStorage.setItem("username", "yossi"); localStorage.setItem("flights", JSON.stringify(flightsArray)); // שליפה: const name = localStorage.getItem("username"); const raw = localStorage.getItem("flights"); const flights = JSON.parse(raw); // חזרה לאובייקט // שליפה עם ערך ברירת מחדל: const theme = localStorage.getItem("theme") || "light"; // מחיקה: localStorage.removeItem("username"); localStorage.clear(); // מחיקת הכל // בדיקת קיום: if (localStorage.getItem("flights")) { // יש נתונים שמורים — אין צורך לטעון מחדש const cached = JSON.parse(localStorage.getItem("flights")); createFlights(cached); } else { // אין נתונים — טעינה מהשרת fetch("./flights.json")... }
דפוס נפוץ — Cache מקומי:
טעינת נתונים מהשרת פעם אחת → שמירה ב-localStorage → בטעינות הבאות שימוש בגרסה המקומית. חוסך בקשות רשת מיותרות ומשפר ביצועים.

Promise Flow

זרימת Promise — תרשים מושגי

Promise מייצג פעולה שתסתיים בעתיד. היא יכולה להסתיים בהצלחה (resolve) או בכישלון (reject). כל שלב בשרשרת .then() מקבל את תוצאת השלב הקודם.

1
fetch("./flights.json") — שליחת בקשה
הדפדפן שולח HTTP GET לשרת. מחזיר Promise. הקוד ממשיך לרוץ.
↓
2
ממתין לתגובה (Pending)
Promise במצב "ממתין". שאר הקוד ממשיך לרוץ במקביל.
↓
3
.then(result =>) — תגובה הגיעה
result = Response object. בדיקת result.ok, החזרת result.json()
↓
4
.then(data =>) — JSON פורסר
data = אובייקט JS מוכן לשימוש. כאן מתבצעים הרינדור והשמירה.
↓ (רק אם יש שגיאה)
!
.catch(err =>) — טיפול בשגיאה
מופעל בשגיאת רשת, CORS, או throw מכל .then לפניו.
.catch בסוף השרשרת תופס שגיאות מכל ה-.then שלפניו — לא רק מהאחרון. זוהי ההסיבה שכותבים catch אחד בסוף ולא לכל then.

תרגול מסכם — מערכת טיסות
